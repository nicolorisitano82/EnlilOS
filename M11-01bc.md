## M11-01b · TLS e Startup Runtime

### Obiettivo

Rendere il thread pointer (`TPIDR_EL0`) corretto e stabile per ogni task EL0,
e costruire il startup C (`crt1/crti/crtn`) che musl si aspetta all'entry point.

### Blocco B1 · TPIDR_EL0 nel contesto task

#### File da modificare

| File | Modifica |
|------|----------|
| `kernel/sched.c` | aggiungere `tpidr_el0` in `sched_task_ctx_t` |
| `kernel/sched_switch.S` | save/restore `TPIDR_EL0` in `sched_context_switch` |
| `kernel/sched.c` | propagare in `fork()` e azzerare in `execve()` |
| `include/sched.h` | aggiungere `sched_task_set_tpidr()` / `sched_task_get_tpidr()` |

#### Modifica a `sched_task_ctx_t` (`kernel/sched.c`)

```c
typedef struct {
    mm_space_t *mm;
    uint64_t    kernel_stack_pa;
    uint64_t    user_entry;
    uint64_t    user_sp;
    uint64_t    argc;
    uint64_t    argv;
    uint64_t    envp;
    uint64_t    auxv;
    uint64_t    tpidr_el0;   /* ← NUOVO: thread pointer user-space */
    uint8_t     is_user;
    uint8_t     resume_from_frame;
    uint8_t     _pad[6];
    int32_t     exit_code;
    uint32_t    parent_pid;
    uint32_t    pgid;
    uint32_t    sid;
    exception_frame_t start_frame;
} sched_task_ctx_t;
```

**Nota:** `sched_tcb_t` è esattamente 64 byte e NON va toccato. `sched_task_ctx_t`
è separato (array `task_ctx[SCHED_MAX_TASKS]`) e può crescere.

#### Modifica a `sched_context_switch` (`kernel/sched_switch.S`)

Dopo `str x2, [x0, #0]` (salva SP di prev) e prima di `ldr x2, [x1, #0]` (carica SP di next),
salvare/ripristinare `TPIDR_EL0`.

Il context switch usa solo `x0`, `x1`, `x2` e i registri callee-saved. I due task
sono identificati da `x0=prev` e `x1=next`. `TPIDR_EL0` va salvato nella struttura
`sched_task_ctx_t`, non nel TCB. Servono due approcci:

**Approccio A (semplice, raccomandato):** salvare/ripristinare nel wrapper C `schedule_locked`.

```c
// In schedule_locked(), PRIMA di sched_context_switch(prev, next):
{
    sched_task_ctx_t *prev_ctx = ctx_of(prev);
    sched_task_ctx_t *next_ctx = ctx_of(next);
    __asm__ volatile("mrs %0, tpidr_el0" : "=r"(prev_ctx->tpidr_el0));
    sched_context_switch(prev, next);
    __asm__ volatile("msr tpidr_el0, %0" :: "r"(next_ctx->tpidr_el0));
}
```

**Approccio B (più robusto):** modificare `sched_context_switch` in assembly
per accettare due puntatori aggiuntivi oppure salvare direttamente i `tpidr`
noti tramite registro dedicato.

Raccomandato **Approccio A**: meno invasivo, tutto il context switch assembly
rimane identico (nessun rischio di rompere il TCB layout a 64 byte).

#### Inizializzazione di `tpidr_el0`

- `sched_task_create_user()`: `ctx->tpidr_el0 = 0ULL`
- `sched_task_fork_user()`: `ctx->tpidr_el0 = parent_ctx->tpidr_el0`
  (il figlio eredita il TP del parent — musl lo re-inizializzerà se necessario)
- `execve()` in `kernel/syscall.c` (`sys_execve`): `ctx->tpidr_el0 = 0ULL`
  (dopo exec, il nuovo processo parte con TP=0; il crt1/musl lo inizializza)

#### Selftest

Aggiungere selftest `tls-tp` in `kernel/selftest.c`:

```c
static int selftest_case_tls_tp(void)
{
    /* Lancia TLSDEMO.ELF che:
     * 1. scrive un valore in TPIDR_EL0
     * 2. chiama sched_yield (syscall SYS_YIELD)
     * 3. legge TPIDR_EL0 e verifica che sia lo stesso
     * 4. chiama fork, nel child verifica che sia uguale al parent
     * 5. exit(0) se tutto ok, exit(1) altrimenti
     */
    uint32_t pid = 0U;
    sched_tcb_t *task;

    if (st_spawn_user_task("tls-tp", "/TLSDEMO.ELF", PRIO_KERNEL, &pid) < 0)
        return -1;
    task = st_wait_task_state(pid, TCB_STATE_ZOMBIE, 3000ULL);
    ST_CHECK("tls-tp", task != NULL, "TLSDEMO.ELF timeout");
    ST_CHECK("tls-tp", st_expect_exit_code("tls-tp", pid, 0) == 0, "TPIDR_EL0 non preservato");
    return 0;
}
```

Aggiungere `user/tls_demo.c` (user-space freestanding che usa `user_svc.h`,
testa save/restore TPIDR_EL0 attraverso yield e fork).

### Blocco B2 · PT_TLS handling nell'ELF loader

musl usa un segmento `PT_TLS` per descrivere il TLS template (dati statici
thread-local). Il loader deve:

1. Riconoscere `PT_TLS` in `elf_load_common()` (`kernel/elf_loader.c`)
2. Allocare il TCB + TLS block nel VAS user-space
3. Copiare il template TLS nel blocco allocato
4. Inizializzare `TPIDR_EL0` con l'indirizzo del TCB

Per musl **statico single-thread** il layout TLS AArch64 è:

```
[TLS data (p_filesz byte)] [TLS bss (p_memsz - p_filesz byte)] [TCB (8-16 byte)]
```

Il thread pointer (TPIDR_EL0) punta **alla fine del blocco TLS** (ovvero all'inizio
del TCB per la variante musl, che usa `tp - sizeof(tcb)` per raggiungere il tcb).

In pratica per musl AArch64 static:

```
tpidr_el0 = indirizzo immediatamente DOPO il blocco TLS data+bss
            (quindi tpidr_el0[-1] è l'ultimo byte del TLS bss)
```

#### Implementazione in `elf_load_common()` / `elf_load_path_exec_dynamic()`

```c
/* In elf_image_t aggiungere: */
typedef struct {
    ...
    uintptr_t tls_addr;    /* VA base del blocco TLS allocato (0 se PT_TLS assente) */
    uintptr_t tpidr_el0;   /* valore iniziale da caricare in TPIDR_EL0 */
} elf_image_t;
```

```c
/* In elf_load_common(), dopo il loop sui PT_LOAD: */
for (uint32_t i = 0; i < ehdr.e_phnum; i++) {
    if (phdrs[i].p_type != PT_TLS) continue;
    uintptr_t tls_align  = phdrs[i].p_align ? phdrs[i].p_align : 1;
    uintptr_t tls_filesz = phdrs[i].p_filesz;
    uintptr_t tls_memsz  = phdrs[i].p_memsz;
    /* Alloca sopra lo stack (o in un range dedicato) */
    uintptr_t tls_block  = ...; /* VA allineata, alloca pagine */
    /* Copia template */
    memcpy((void *)tls_block, (void *)(phdrs[i].p_vaddr + bias), tls_filesz);
    memset((void *)(tls_block + tls_filesz), 0, tls_memsz - tls_filesz);
    /* AArch64 TLS variant 1 (musl): tp = tls_block + tls_memsz */
    out->tpidr_el0 = tls_block + tls_memsz;
    break;
}
```

**Nota:** Per `M11-01` (statico, single-thread) il blocco TLS può essere allocato
come una pagina anonima nel VAS, piazzata sotto lo stack user-space (es. `user_sp - PAGE_ALIGN(tls_memsz + 16)`).

Quando si lancia il task (`sched_task_create_user` / `sched_enter_user`), passare
`tpidr_el0` e scriverlo in `ctx->tpidr_el0` che poi viene caricato in `TPIDR_EL0`
dal salvataggio/ripristino del context switch (vedi Blocco B1).

Per retrocompatibilità: se `PT_TLS` assente, `tpidr_el0 = 0`.

### Blocco B3 · auxv: aggiungere AT_RANDOM e AT_UID/GID

L'array `auxv` nel loader (`kernel/elf_loader.c`, intorno alla riga 391) ha oggi 8 slot.

Da aggiungere:

```c
// Dentro elf_build_stack():
auxv[auxc++] = (elf64_auxv_t){ ELF_AT_RANDOM, random_page_va };   /* 16 byte casuali */
auxv[auxc++] = (elf64_auxv_t){ ELF_AT_UID,    0ULL };
auxv[auxc++] = (elf64_auxv_t){ ELF_AT_EUID,   0ULL };
auxv[auxc++] = (elf64_auxv_t){ ELF_AT_GID,    0ULL };
auxv[auxc++] = (elf64_auxv_t){ ELF_AT_EGID,   0ULL };
```

`AT_RANDOM`: allocare una pagina anonima, riempirla con 16 byte casuali da
`CNTPCT_EL0 ^ cap_salt` (già usato in `cap.c`), passarne la VA. Il pointer
punta ai 16 byte, non all'intera pagina.

Espandere l'array `auxv[8]` → `auxv[16]` in `elf_build_stack()`.

Costanti mancanti da aggiungere in `kernel/elf_loader.c` o `include/elf.h`:

```c
#define ELF_AT_RANDOM  25
#define ELF_AT_UID     11
#define ELF_AT_EUID    12
#define ELF_AT_GID     13
#define ELF_AT_EGID    14
```

### Blocco B4 · crt1 / crti / crtn

Il `crt1` è il file oggetto contenente `_start`, il vero entry point dei binari ELF
statici musl. Oggi EnlilOS entra con `x0=argc, x1=argv, x2=envp, x3=auxv`.

Per musl AArch64 statico, `_start` deve:

1. Impostare `sp` (già impostato dal kernel via `SP_EL0`)
2. Chiamare `__libc_start_main(main, argc, argv)`

musl fornisce già il suo `crt1.c` / `crt1.S` per `aarch64`. Il nostro adattamento
è minimo: assicurarsi che il kernel passi gli argomenti nel modo che musl `_start` si aspetta.

**ABI musl AArch64 `_start`:**
```
sp → [argc] [argv[0]] ... [argv[argc-1]] [0] [envp[0]] ... [0] [auxv pairs] [0,0]
```

Questa è **esattamente** la stessa ABI che EnlilOS già costruisce in `elf_build_stack()`.
Non serve modificare il kernel — basta puntare l'ELF entry al `_start` di musl.

**`crti.o` / `crtn.o`:** contengono le sezioni `.init` e `.fini`. Per la v1 statica
possono essere stub vuoti:

```asm
/* crti.S */
.section .init
.global _init
_init:
.section .fini
.global _fini
_fini:
```

```asm
/* crtn.S */
.section .init
    ret
.section .fini
    ret
```

Questi vanno costruiti per target `aarch64-elf` con il GCC host AArch64 già disponibile.

### Blocco C1 · Strategia toolchain

**Raccomandazione**: wrapper GCC + sysroot dedicato, senza musl-cross-make.

`aarch64-elf-gcc` è già presente nella toolchain di EnlilOS. musl può essere
compilata da esso una volta che ha il proprio sysroot.

#### Layout sysroot

```
sysroot/
  include/          ← header musl (copiati da third_party/musl/include/)
  lib/
    crt1.o
    crti.o
    crtn.o
    libc.a
    libm.a
  usr/              ← symlink o copia di include/ e lib/ (per compat)
```

#### Wrapper `aarch64-enlilos-musl-gcc`

Script shell:

```sh
#!/bin/sh
exec aarch64-elf-gcc \
    --sysroot="$(dirname "$0")/../sysroot" \
    -static \
    -nostdinc \
    -isystem "$(dirname "$0")/../sysroot/include" \
    "$@"
```

Posizionato in `toolchains/bin/aarch64-enlilos-musl-gcc`.

#### Compilazione di musl

```sh
cd third_party/musl
./configure \
    --target=aarch64 \
    --prefix=/sysroot \
    CC="aarch64-elf-gcc" \
    CFLAGS="-mcpu=cortex-a72 -ffreestanding" \
    --disable-shared
make -j$(nproc)
make install DESTDIR=$(pwd)/../../sysroot
```

Il configure di musl supporta cross-compilation a target `aarch64`.
`--disable-shared` produce solo `libc.a`.

### Blocco C2 · Patch musl per EnlilOS

Il delta rispetto a musl upstream deve essere **minimo e concentrato**.

#### File da patchare

**`arch/aarch64/syscall_arch.h`** (mapping numeri syscall):

Oggi musl usa numeri Linux AArch64. EnlilOS usa gli stessi numeri per molte
syscall (SYS_read=63, SYS_write=64, ecc. — standard Linux AArch64). I gap
sono nei numeri personalizzati EnlilOS (80–159).

Per `M11-01c` (smoke test minimi), musl userà solo:

```
read=63, write=64, open=56, close=57, fstat=80, exit=93, exit_group=94,
mmap=222, munmap=215, brk=214, getpid=172, kill=129, sigaction=134,
fork=1079 (clone), waitpid=260 (wait4)
```

Questi numeri **coincidono già** con Linux AArch64 → zero patch sui numeri
per i primitivi di base.

I numeri EnlilOS personalizzati (es. `lseek=8` vs Linux `lseek=62`) richiedono
un piccolo override. Aggiungere in `arch/aarch64/syscall_arch.h` EnlilOS:

```c
/* Override per EnlilOS — numeri che divergono da Linux AArch64 */
#undef SYS_lseek
#define SYS_lseek 8         /* EnlilOS: SYS_LSEEK */
```

**`src/errno/__errno_location.c`** e **`src/env/__init_tls.c`**:

Questi usano `TPIDR_EL0` — funzionano già su AArch64 se il TP è inizializzato.
Non richiedono patch se il kernel gestisce correttamente `PT_TLS` (vedi B2).

**`include/bits/syscall.h`** e **`include/bits/alltypes.h`**:

`alltypes.h` per AArch64 è già incluso in musl. Non serve patch per la v1.

**`src/linux/clone.c`** (per fork):

musl usa `clone(SIGCHLD, 0)` per `fork()`. EnlilOS implementa `fork()` come
syscall separata (nr 1079 = Linux `clone`). Verificare che il numero coincida
o aggiungere un override minimo.

## Rischi

| Rischio | Probabilità | Mitigazione |
|---------|-------------|-------------|
| musl usa `AT_RANDOM` nel bootstrap init | Alta | Implementare B3 prima di tentare il primo run |
| `PT_TLS` assente → errno corrotto | Alta | B2 prima di qualsiasi test musl |
| `fork()` musl chiama `clone` con flag diversi da quelli EnlilOS | Media | Verificare `src/linux/clone.c` musl e il mapping del numero syscall |
| `mmap` anonima per TLS block non gestita correttamente | Media | Usare `phys_alloc_page` direttamente dal loader, come per lo stack |
| Dimensione TLS block variabile a seconda del binario musl linkato | Bassa | Arrotondare sempre a multiplo di pagina |
| `sched_task_ctx_t` overflow se si aggiunge troppo | Bassa | ctx è separato dal TCB, può crescere liberamente |
